# Task 3 開發計畫書：業界實務導向的動態機器學習展示網站

## 1. 作業需求判定

原始任務明確指定「開發動態學習網頁（Next.js + FastAPI）」，因此下列項目屬於確定需求：

- 必須是可互動的網頁，而不是只有靜態文章。
- 技術架構採 Next.js 前端與 FastAPI 後端，屬於前後端分離。
- 網站內容需延續 Task 1 的十大模型資訊圖表與 Task 2 的實務教材。
- 後續需能銜接 Task 4 的 AI 助理。

作業紀錄沒有明文要求一定要做成完整線上課程、帳號系統、學習進度、測驗平台或 Python 線上程式執行器。「學習導向」應理解為幫助使用者理解模型，而不是必須實作傳統 LMS。

因此 Task 3 的正式定位調整為：

> 以台灣產業情境為主軸，透過模型選擇、參數模擬與業界流程展示，幫助使用者理解模型如何被選擇、調整與應用。

## 2. 產品目標

網站優先回答四個實務問題：

1. 這個業界問題應該先使用哪種模型？
2. 重要參數改變時，模型行為會如何改變？
3. 指標變好是否代表方案真的能在業界使用？
4. 訓練、部署與監控時有哪些常見風險？

網站不是用來背誦公式，也不以大量 Python 程式碼作為主要內容。若作業未要求，模型頁不預設展示 Python example。

## 3. 正式功能範圍

### 3.1 首頁：Industry ML Workbench

首頁不是一般課程首頁，而是業界工作台，提供：

- 十大模型快速入口。
- 五種產業情境：半導體與製造、金融風控、智慧醫療、零售電商、房地產與需求預測。
- 模型選擇器入口。
- 參數實驗室入口。
- Task 1 資訊圖表預覽。
- Task 2 PDF 教材下載。
- GitHub Logo 與 repository 連結：<https://github.com/dec591nyc/Machine-Learning-Study>。
- 預留 AI 助理入口，但 Task 3 不實作真正對話。

### 3.2 模型實務頁

每個模型頁固定呈現：

- 適合解決的業界問題。
- 輸入資料、輸出結果與必要前處理。
- 重要參數及其影響。
- 主要評估指標。
- 何時不應使用。
- 台灣產業模擬情境。
- 從資料到部署的流程圖。
- 若該模型適合視覺化，連至參數實驗室。

頁面不加入沒有作業必要性的長篇 Python 範例。技術名詞可用短小 pseudo workflow 或 API 輸入輸出說明取代。

### 3.3 模型選擇器

使用者依序選擇：

- 任務是數值預測、分類、分群或降維。
- 資料型態是表格、文字、影像或高維資料。
- 樣本量與特徵量級。
- 是否要求高可解釋性。
- 是否有類別不平衡、即時延遲或運算限制。

FastAPI 依透明規則回傳：

- 建議的基準模型。
- 第二個比較模型。
- 選擇理由。
- 不適用風險。
- 建議評估指標。
- 建議進入的參數模擬。

推薦規則必須可測試、可解釋，不在 Task 3 使用生成式 AI 猜測答案。

### 3.4 模型比較器

最多比較三個模型，欄位包括：

- 任務類型與資料需求。
- 是否需要標準化。
- 可解釋性。
- 訓練與推論成本。
- 適合資料規模。
- 過度擬合風險。
- 重要參數。
- 業界部署注意事項。

桌面版使用資料表；手機版改為逐項比較卡，避免水平捲動。

## 4. 參數實驗室

參數實驗室是網站的核心互動功能。目的不是讓使用者寫程式，而是讓使用者直接看見參數如何改變模型。

### 4.1 互動結構

```text
業界情境與資料說明
        ↓
參數控制面板 ─ 即時模型圖表 ─ 變化原因與業界解讀
        ↓
訓練／驗證指標與風險提示
```

每次調整參數後：

1. 前端等待短暫 debounce，避免滑桿連續觸發大量請求。
2. FastAPI 使用固定種子與內建小型資料集重新計算。
3. 回傳圖表資料、訓練與驗證指標、複雜度和解釋代碼。
4. 前端更新圖表並顯示「發生什麼、為什麼、業界代表什麼」。

### 4.2 正式保留的六種模擬

只實作能穩定呈現參數效果、視覺辨識度高的模型：

| 模型 | 可調參數 | 顯示結果 | 實務理解 |
| --- | --- | --- | --- |
| 線性／Ridge 回歸 | `alpha`、資料雜訊 | 擬合線、殘差、MAE | 正則化、偏差與穩定性 |
| 邏輯回歸 | `C`、分類門檻 | 機率曲線、混淆矩陣 | 門檻如何改變 precision/recall |
| 決策樹 | `max_depth`、`min_samples_leaf` | 決策區域、訓練／驗證差距 | 欠擬合與過度擬合 |
| SVM | `C`、`gamma` | 邊界、margin、支持向量 | 邊界複雜度與泛化 |
| KNN | `k`、距離權重 | 鄰居與分類區域 | 局部決策與維度風險 |
| K-Means | `k`、初始化 | 群組、質心、silhouette | 群數不是越多越好 |

### 4.3 縮減或捨棄的模擬

以下功能不強行實作：

- 隨機森林即時逐棵樹動畫：視覺資訊量低且效能成本高，改以模型比較、特徵重要度與穩定度圖表示範。
- 單純貝氏完整互動實驗：參數變化不易被一般使用者直觀理解，改用文字分類機率案例卡。
- PCA 3D 高成本互動：若 2D 投影與解釋變異圖效果清楚才加入；否則只保留靜態可切換成分圖。
- 神經網路在瀏覽器或 API 即時訓練：結果受隨機性、時間與硬體影響，容易造成不穩定體驗。改用預先計算的學習曲線，展示學習率、模型容量與過擬合概念。
- 任意資料上傳與任意程式碼執行：涉及安全、資料品質與執行時間，不納入 Task 3。
- 帳號、登入、資料庫與跨裝置進度：作業未要求，會分散核心開發資源，先不實作。
- 傳統課程進度、徽章與大型題庫：不是硬性需求，改以瀏覽紀錄和最近查看模型提供輕量導覽。

若 PCA 的 2D 互動在實作驗證後不清楚，也直接移除，不為湊滿十個實驗而保留。

## 5. 業界情境模擬

網站使用「模擬情境」而非宣稱特定企業實際採用某模型。

### 半導體與製造

- 製程參數與良率風險。
- 設備異常預警。
- 模型門檻對工程師每日覆核量的影響。

### 金融風控

- 違約或詐欺風險排序。
- 分類門檻對漏報、誤報與人工審查量的影響。
- 可解釋性與機率校準的重要性。

### 智慧醫療

- 風險分層與人工覆核。
- 高 recall 與誤報成本的取捨。
- 明確標示僅為教育模擬，不提供醫療決策。

### 零售電商

- 會員分群、需求預測與流失風險。
- 分群是否能形成不同營運策略。

### 房地產與需求預測

- 價格估計、區域誤差與模型穩定性。
- 線性基準與非線性模型的取捨。

## 6. Logo 與品牌視覺策略

網站可大量使用 Logo 增加技術辨識度，但必須避免造成未經證實的企業合作或模型採用暗示。

### 6.1 可使用的技術 Logo

- Next.js

- scikit-learn
- NumPy
- pandas

來源優先使用 Simple Icons 或各專案官方 SVG。所有 Logo：

- 提供 `aria-label` 或可讀文字。
- 不用 Logo 取代必要的文字標籤。
- 不在模擬產業案例上放未驗證的企業 Logo。

### 6.2 GitHub 連結

導覽列右側與頁尾都提供 GitHub Logo 連結：

<https://github.com/dec591nyc/Machine-Learning-Study>

- 使用官方 GitHub mark。
- 新分頁開啟。
- 加上可見文字「View on GitHub」或中文輔助標籤。

### 6.3 一般介面圖示

操作、導航、警告與狀態使用 Lucide SVG，不使用 emoji 作為結構圖示。Logo 表示技術品牌，Lucide 表示介面動作，兩者不混用。

## 7. UI/UX 設計系統

### 7.1 設計方向

採用 Base Web 的資料密集與清晰控制理念，但不直接套用完整 Base Web 視覺框架。實作組合：

- shadcn/ui 與 Radix／Base UI：無障礙互動元件。
- Tailwind CSS：品牌樣式和響應式版面。
- TanStack Table：比較資料表。
- Plotly.js：決策邊界、分群與科學圖形。
- Recharts：一般指標與趨勢圖。
- Lucide：介面圖示。

### 7.2 色彩

只提供亮色模式，不開發暗色模式。

- Primary：`#C2410C`
- Accent：`#EA580C`
- Strong text：`#292524`
- Secondary text：`#57534E`
- Background：`#FFF8F1`
- Surface：`#FFFFFF`
- Soft orange：`#FFEDD5`
- Border：`#E7D8CC`

深橘色用於主要操作、目前選項和重要圖表線條；錯誤、成功與警告保留各自語意色，避免所有狀態都變成橘色。

### 7.3 版面

- 桌面：左側情境／模型導航，中間圖表，右側實務解讀。
- 平板：控制面板可收合，圖表與解讀上下排列。
- 手機：單欄，參數控制置於圖表前，不產生水平捲動。
- 長文內容維持 60–75 字元行寬。
- 控制元件至少 44 px 點擊範圍。
- 圖表資訊不只依賴顏色，需搭配線型、符號與文字。

## 8. 前後端架構

```text
Browser
  │
  ▼
Next.js Web
  │ /api/*
  ▼
FastAPI
  ├─ recommendation rules
  ├─ bounded model experiments
  ├─ model metadata
  └─ AI context contract
```

Next.js 負責：

- 頁面、互動、Logo、圖表、響應式與無障礙。
- 輕量瀏覽紀錄與使用者目前選擇。
- 將目前頁面與實驗狀態整理成 AI context。

FastAPI 負責：

- 模型推薦規則。
- 固定資料集與受限參數的模型計算。
- 統一回傳圖表資料與解釋代碼。
- 後續 Task 4 可使用的 context schema。

開發時前後端分離，但透過 Next.js rewrite 將 `/api/*` 代理到 FastAPI，讓瀏覽器端維持單一來源。

## 9. AI 助理銜接

Task 3 不直接串接 AI 模型，但每個頁面建立標準 context：

```json
{
  "page": "parameter-lab",
  "industry": "manufacturing",
  "model": "decision-tree",
  "parameters": {
    "max_depth": 8,
    "min_samples_leaf": 2
  },
  "metrics": {
    "train_accuracy": 0.99,
    "validation_accuracy": 0.82
  },
  "diagnostic_codes": ["possible_overfitting"]
}
```

Task 4 的 AI 助理將能根據這些內容解釋：

- 為什麼目前參數可能過度擬合。
- 下一個值得嘗試的參數。
- 指標變化對業界流程的含義。
- 目前模型是否符合使用者選擇的產業限制。

Task 3 只提供「詢問 AI」入口與 context contract，不放置無法使用的假聊天視窗。

## 11. 開發順序與淘汰閘門

1. 建立 Next.js + FastAPI 前後端骨架與單一開發指令。
2. 建立深橘色設計 token、Logo 系統及應用程式外框。
3. 將 Task 1、Task 2 內容整理成模型與產業結構化資料。
4. 完成首頁、模型實務頁、GitHub 連結及 PDF／資訊圖表入口。
5. 完成透明規則式模型選擇器與比較器。
6. 先製作決策樹與邏輯回歸兩個模擬原型。
7. 通過「效果是否清楚、互動是否穩定、手機是否可用」驗收後，再擴充另外四個模擬。
8. 若原型無法清楚傳達參數效果，立即縮減或捨棄，不擴充相同類型功能。
9. 建立 AI context contract 與預留入口。
10. 完成 API 測試、核心元件測試、瀏覽器操作與響應式視覺 QA。

### 模擬功能驗收條件

每個模擬必須同時滿足：

- 參數改變後的圖表差異肉眼可辨識。
- 中文說明能指出原因，而非只顯示分數。
- 同一參數與固定種子能重現結果。
- 一般開發機器可在合理時間內回應。
- 375 px 手機畫面仍可操作。
- 失敗時有清楚錯誤訊息與重試方式。

未達成上述條件的模擬不進入正式版本。
